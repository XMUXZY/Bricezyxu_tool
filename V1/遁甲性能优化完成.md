## 📊 优化前后对比

### ❌ 优化前（慢）
- **数据结构**：列表（249条数据）
  ```python
  self.LEVEL_DATA = [
      ('黄阶', 1, 55, 1, 1),
      ('黄阶', 2, 165, 1, 2),
      ...  # 249条数据
  ]
  ```

- **查询方式**：遍历查找
  ```python
  def _get_level_cost(self, grade, level):
      for data in self.LEVEL_DATA:  # 每次遍历249条
          if data[0] == grade and data[1] == level:
              return data[3]
      return 0
  ```

- **时间复杂度**：`O(n)` - 每次查询平均需要遍历 125 次
- **总查询次数**：计算一次可能需要数百次查询
- **总时间复杂度**：`O(n × m)` = O(249 × 200) ≈ **50,000次比较**

### ✅ 优化后（快）
- **数据结构**：字典索引
  ```python
  self.LEVEL_DATA = {
      ('黄阶', 1): (55, 1, 1),      # (能力值, 单级消耗, 累计消耗)
      ('黄阶', 2): (165, 1, 2),
      ...  # 249个键值对
  }
  ```

- **查询方式**：直接查找
  ```python
  def _get_level_cost(self, grade, level):
      data = self.LEVEL_DATA.get((grade, level))  # O(1)直接获取
      return data[1] if data else 0
  ```

- **时间复杂度**：`O(1)` - 直接通过键查找，1次查询
- **总时间复杂度**：`O(m)` = O(200) ≈ **200次查询**

## 🔧 具体优化内容

### 1. 数据结构改造
```python
# 之前：列表（需要遍历）
data = []
data.append(('黄阶', lv, ability, cost, cumulative))

# 现在：字典（直接索引）
data_dict = {}
data_dict[('黄阶', lv)] = (ability, cost, cumulative)
```

### 2. 查询方法优化
**优化了3个核心查询方法：**
- `_get_level_cost()` - 获取单级消耗
- `_get_cumulative_cost()` - 获取品阶内累计
- `_get_ability()` - 获取能力值

```python
# 全部从 O(n) 遍历优化为 O(1) 字典查找
data = self.LEVEL_DATA.get((grade, level))
```

### 3. 内存占用
- **几乎没有增加**（只是数据结构变化）
- 列表：~249个元组
- 字典：~249个键值对
- 增量：仅字典索引的额外开销（可忽略）

## 🎯 技术细节

### 优化原理
**哈希表查找 vs 线性查找**

| 操作 | 列表（线性） | 字典（哈希） |
|------|-------------|-------------|
| 查找 | O(n) | O(1) |
| 插入 | O(1) | O(1) |
| 空间 | O(n) | O(n) + 额外哈希表 |

### 适用场景
✅ **适合字典优化的场景**：
- 频繁的精确查找（通过品阶+等级查找）
- 数据量中等（249条）
- 键值对明确（品阶、等级唯一确定一条数据）

❌ **不适合字典优化的场景**：
- 需要范围查询
- 需要按顺序遍历
- 数据量极小（<10条）

### 为什么遁甲计算器特别适合优化？
1. **查询频繁**：每级都要查询能力值、消耗、累计
2. **查询随机**：不是顺序查找，而是跳跃式查询
3. **数据量适中**：249条数据，列表遍历慢，字典索引快
4. **键值对明确**：(品阶, 等级) 唯一确定一条数据
